Dominator

One unified, telemetry-free editor-productivity suite for Unity — colored hierarchy & folders, tabs & favorites, a command palette, themes, cleanup, notes, backup and more. One install, one settings panel, one license. No scripting required, nothing phones home.

Unity 2021.3+ Editor-only · zero player cost Telemetry-free No-code, click-to-do
ⓘ
One suite, not a bag of tools. Every module shares one rule engine, one palette and one committed config. A single rule you author once styles the Hierarchy and the Project window and feeds Notes/Console/Cleanup — that integration is the product, and it is what a pile of separate assets can't copy.
✅
Telemetry-free by construction. Network access is confined to two audited, opt-in files (BYO-key AI and opt-in remote logging) and CI fails the build if anything else reaches the network. Offline by default, no account, no tracking.

⤓ Installation

Import once. There is nothing to configure to start — the suite is live the moment it compiles.

Import from the Asset Store
In Window → Package Manager → My Assets, download and import Dominator. It lands under Assets/Dominator/.
Let it compile
On first import a Welcome window greets you with one-click links to the docs, the demo scene, and your first organizer. (Re-open it any time from Tools → Dominator → Welcome.)
Open the settings
Project Settings → Dominator is the single, searchable home for every module's options. Per-user preferences live under Preferences → Dominator.
That's it
Your Hierarchy and Project window are already enhanced. Add a rule or drop an organizer to see decoration appear.

Requirements

  • Unity 2021.3 or newer (developed and validated on Unity 6000.0).
  • Editor-only. All logic lives in Editor assemblies and is stripped from player builds — zero runtime cost, zero footprint in your shipped game.
  • No internet connection, account, or external service required.

⚡ Quick Start

Author one rule and watch it flow through every surface — about a minute, no code.

Open the rule editor
Tools → Dominator → Rules → Open Rule Editor (Ctrl+Alt+U), or use New Rule….
Make a rule
e.g. Name starts with NPC_ → a red tint and an icon. Choose whether it applies to the Hierarchy, the Project window, or both.
Watch it apply everywhere
Every matching GameObject row and matching asset/folder is decorated instantly — the same rule, across both surfaces, across every scene.
Add a hierarchy organizer
GameObject → Dominator → Header / Divider / Folder drops a labelled organizer row to structure your scene. Configure it in the Inspector (it has tooltips).
Find anything
Ctrl+K opens the command palette — search assets, objects, settings, commands and favorites in one box.
💡
Everything is undoable. Rules, organizers, theme changes and migrations are all wired to Unity's Undo — one Ctrl+Z reverts the whole action.

⚙ How It Works

The integration moat: one rule, one palette, one config — flowing through every module.

Unlike a collection of separate assets, Dominator resolves every visual decision through one shared rule engine and one palette. A pattern rule (by name, type, tag, layer, path or asset type) is evaluated once and consumed identically by the Hierarchy decorator and the Project-folder resolver, so the icon you assign is the same icon everywhere — by construction, not by copying settings between tools.

🧮 Unified rule engine

One committed DominatorRuleSet drives hierarchy rows and folder faces. Pattern-based, so a rule auto-applies across every scene and prefab.

🎨 One visual language

A single palette and theme token set resolves identically into every surface — no per-module color tables to keep in sync.

👥 Team-config-sync

All rules, palette, theme and quick-access config is committed, merge-friendly text. One git pull = an identical studio setup.

🔧 No-Code / Editor Tools

Dominator is built to be driven by clicking. Every primary capability is reachable from a menu, the settings panel, or the Inspector — no scripting needed.

✅
A non-coder is the target user. Open the settings, pick options, assign a rule, drop an organizer — and get a correct, visible result with no script. The full scripting API (below) is there for power users, but nothing requires it.

First-run Welcome window

On first import a Welcome window opens once (it never nags again, and is suppressed in batch/CI). It offers one-click buttons to open this documentation, open the interactive demo scene, create your first organizer, and start the guided tour. Re-open it any time from Tools → Dominator → Welcome.

The unified settings panel

Project Settings → Dominator is one searchable page with a section rail covering every module — general, accessibility, theming, localization, AI, licensing and more. Per-user options live under Preferences → Dominator so they never dirty the committed team config.

⚙ Settings & Team Sync

Committed where it should be shared, per-user where it shouldn't.

  • Project (committed): rules, palette, theme, folder styles and quick-access live as text under ProjectSettings/Dominator/ — merge-friendly and GlobalObjectId-keyed, so they survive refactors and travel with the repo.
  • Per-user (local): personal preferences and onboarding state live under your user preferences folder and never dirty the committed config or cause merge conflicts.
  • Import / export: Tools → Dominator → Export Config… / Import Config… round-trips the whole setup as reviewable JSON.
  • Reset: Reset All Settings… returns to defaults; every change is Undo-wired and snapshot-protected (see Backup).
⚠
License tiers. Some advanced features are gated by tier (Free / Pro / Studio). Activate from Tools → Dominator → License. Owned features never re-lock on upgrade.

📝 Inspector & Serializer

A better Inspector, and a serializer with a no-lock-in exit ramp.

  • 20+ attributes (buttons, groups, tabs, foldouts, validation, conditional show/hide, sliders, progress bars, dropdowns, info boxes) — pure metadata in the runtime assembly, drawers in the editor.
  • A validation scanner that surfaces missing references and broken setups with one-click focus.
  • Serialize-anything (dictionaries, polymorphic SerializeReference) with a safe migration path — and a documented exit ramp so you are never locked in.
  • Coexists with Odin; opt in per type.

🎨 Theming & Accessibility

One visual language across the whole suite, accessible by default.

  • Theme editor (Tools → Dominator → Theming → Theme Editor): one committed theme file styles every surface from one set of roles — not per-window chrome.
  • Built-in themes (Stock, Studio Dark, Studio Light, High-Contrast) plus your own, importable/exportable as .dominatortheme.json.
  • Accessibility: colorblind-safe palettes with redundant icons/labels, WCAG high-contrast theme, reduce-motion, and a one-click accessibility audit.
  • Color-vision preview (Protanopia / Deuteranopia / Tritanopia) to check your scheme.

🧹 Cleanup & Backup

Find dead weight safely, and never lose your config.

Asset Cleanup & Dependencies

  • Unused / duplicate asset detection that is runtime-reference-aware — it understands Resources.Load, reflection and Addressables-style string paths that naive scanners miss.
  • Quarantine, not delete: candidates move to a quarantine folder with a reason and a one-click undo — nothing is destroyed.
  • Reverse-reference ("find usages") graph, also powering honest find-usages elsewhere in the suite.

Backup & Recovery

  • Versioned local snapshots with diff and restore, freeze-aware non-blocking saves, and crash recovery.
  • Undo-integrated recovery of the suite's committed config — a bad git checkout can't silently wipe hours of organization.

📦 Migration & Share Packs

Bring your existing setup in, and share your presets out.

Migration importers

The Tools → Dominator → Migration → Import Wizard… imports your existing setup from popular tools (vHierarchy 2, vFolders 2, Rainbow Folders 2, Rainbow Hierarchy 2, and more). Import is a non-destructive dry run first: you review a diff with a completeness % and a High/Medium/Low confidence band before applying. Apply is idempotent and Undo-wired; your original third-party config is never touched.

Marketplace share packs

  • Export a theme and/or a rule pack as a portable .dominatorpack.json (or a rule-only .dominatorruleset.json) from Tools → Dominator → Share → Export Share Pack….
  • Packs are deterministic (byte-stable for the same input), content-hashed (a tampered pack is detected and refused), and forward-tolerant of newer schema versions.
  • Import is non-destructive and content-deduped: re-importing the same pack is a safe no-op, and imported rules get fresh ids so a shared pack can never clobber a local rule. One shared rule then flows through every surface.

🤖 AI Assist

Optional, bring-your-own-key, offline by default — and it writes into the same unified rule engine.

  • Turn natural language into valid Dominator rules, parse NL commands for the palette, and get icon/color suggestions — from Tools → Dominator → AI.
  • Bring-your-own-key and offline-by-default: an offline heuristic path works with no network at all. If you opt in with a key, egress is confined to a single audited transport file and is mocked in CI.
  • Because AI output is written into the existing DominatorRuleSet, it inherits cross-surface reach and team-sync for free.
🔒
AI is entirely optional and off until you configure a key. With no key, Dominator makes zero network calls.

📙 Scripting API

Everything the menus do is also available to power users in code (the no-code paths are additive, never a replacement).

Author a rule

using Dominator.Editor.Core.Api;

var rule = DominatorRule.CreateDefault("Enemies");
rule.scope   = RuleScope.Both;        // Hierarchy + Project
rule.field   = MatchField.Name;
rule.mode    = MatchMode.StartsWith;
rule.pattern = "NPC_";
rule.backgroundColor = new Color(0.9f, 0.2f, 0.2f);

DominatorRuleSet.instance.Add(rule);   // committed + Undo-wired; flows to every surfaceC#

Create a hierarchy organizer

using Dominator.Editor.Modules.Hierarchy;

GameObject header = OrganizerFactory.CreateHeader("Gameplay");
GameObject folder = OrganizerFactory.CreateFolder("UI");   // Undo-correct, selects + pingsC#

Export / import a share pack

using Dominator.Editor.Core.Packaging;

var pack = DominatorSharePack.CreatePack("Studio Preset", author, desc, version, theme, rules);
DominatorSharePack.Export(pack, path, out var error);     // deterministic, content-hashed

var back = DominatorSharePack.Parse(File.ReadAllText(path), out _);
if (DominatorSharePack.VerifyIntegrity(back)) { /* safe to apply */ }C#

💡 Troubleshooting

Common questions, quick answers.

I don't see any decoration in the Hierarchy / Project window

Decoration is driven by rules. Open Tools → Dominator → Rules → Open Rule Editor and add a rule (e.g. name contains a string), or set a folder color directly via Assets → Dominator → Set Folder Color. Confirm theming is enabled in Project Settings → Dominator.

The Welcome window didn't appear

It opens only once per user (and is suppressed in batch/CI). Re-open it any time from Tools → Dominator → Welcome.

I can't add the Organizer component from "Add Component"

That's intentional — the organizer marker is authored, not hand-added. Create one from GameObject → Dominator → Header / Divider / Folder; then configure it in the Inspector.

Does removing the suite leave junk behind?

No. Run Tools → Dominator → Prepare for Removal… first — it cleans up organizer markers and committed config. The runtime marker is inert and harmless even if left in place.

A migration import didn't bring in everything

The import wizard shows a completeness % and a confidence band before you apply. Anything unresolved is listed so you can map it manually; your original third-party config is never modified.

❓ FAQ

The questions buyers ask most.

Does anything ship in my player build?

Only one inert marker component (DominatorOrganizer) with zero Awake/Update and zero runtime cost. All logic is in Editor assemblies and is stripped from players.

Does it phone home or collect telemetry?

No. Network access is confined to two audited, opt-in files (BYO-key AI and opt-in remote logging), and CI fails the build if anything else reaches the network. Offline by default.

Do I have to write code?

No. Every primary capability is reachable from a menu, the settings panel, or the Inspector. The scripting API is for power users and is purely additive.

Is my config shareable with my team?

Yes — it's committed text under ProjectSettings/Dominator/. Commit it, and a teammate's git pull gives them the identical setup. You can also export portable share packs.

Which Unity versions are supported?

Unity 2021.3 and newer, validated headlessly on 2021.3, 2022.3 and 6000.0. A UI-Toolkit hierarchy backend is included for newer editors.

Does it work with Odin / other inspector tools?

Yes — the inspector enhancements coexist with Odin and are opt-in per type.
