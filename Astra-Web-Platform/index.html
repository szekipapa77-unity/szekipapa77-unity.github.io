Astra Web Platform

Embed real web pages in Unity — a managed browser runtime with platform-native adapters (WebGL iframe, Windows WebView2, and scaffolded WKWebView / Android WebView / WebKitGTK), a JavaScript bridge, a security policy layer, profiles, and diagnostics. Add a component, set a URL, press Play.

Unity 6000.0+ No-code component WebGL + Windows validated Bridge · Policy · Profiles
ⓘ
Two ways to drive it. Add the Astra Web View component and configure it entirely in the Inspector (no scripting), or call the managed runtime API (AstraWebRuntime, BrowserManager, BrowserEngineRegistry) from code. The two paths coexist — the no-code auto-open defers to code when you drive the component yourself.
✅
Validated where it can be. The WebGL iframe adapter and the Windows WebView2 adapter are independently verified to render a real live page (example.com) in a player build. macOS/iOS/Linux/Android adapters ship as build-ready scaffolding and are validated on their own hardware. See Engines & Platforms.

⤓ Installation

Import once, open the Welcome window, add a component.

Import from the Asset Store
In Window → Package Manager → My Assets, download and import Astra Web Platform. It lands at Assets/AstraWebPlatform/.
The Welcome window opens automatically
On first import the Welcome / Help window appears with big buttons to open the docs, the demo scene, and create your first web view. Re-open it any time from Tools → Astra Web Platform → Welcome.
Add a web view
GameObject → Astra Web Platform → Astra Web View (or Add Component → Astra Web Platform → Astra Web View).
Set a URL and press Play
In a player build the platform-native engine renders the page; in the Editor the Mock engine is used (expected). For Windows, the WebView2 Evergreen runtime must be installed.

Requirements

  • Unity 6000.0 or newer.
  • Windows player: the WebView2 Evergreen runtime (preinstalled on Windows 11; bundled with modern Edge).
  • macOS / iOS / Linux / Android native adapters require their platform binaries — see the cross-platform build guide and Engines & Platforms.

⚡ Quick Start (No Code)

A real web page in your scene without writing a line of code.

Create the web view
GameObject → Astra Web Platform → Astra Web View. A sample-populated object drops into the scene (URL https://example.com/, Engine Automatic, Open On Start).
Set the URL & engine in the Inspector
Type your URL. Leave Engine on Automatic to use the best engine for the platform, or pick a specific one. The inspector shows whether that engine is available in the current build target.
(Optional) Assign a Settings asset
For a custom security policy/size, create Assets → Create → Astra Web Platform → Settings and drag it into the Settings Asset slot. Without one, a sensible default policy that allows your typed URL is used.
Press Play
The page opens through the resolved engine. In a Windows player it renders via WebView2; in a WebGL build via the iframe adapter; in the Editor the Mock engine runs (no visible page, but the full pipeline executes).
💡
Driving it from code? Call SuppressAutoOpen() (or set Open On Start off) so the no-code auto-open does not run on top of your own OpenAsync() call. See Scripting API.

⚙ How It Works

From a URL in the Inspector to a rendered page in one build pass.

Inspector fields → Engine factory → Policy check → BrowserManager → Native adapter → Live page
  1. The Astra Web View component reads its serialized fields (URL, engine, size, policy/profile assets).
  2. AstraBrowserEngineFactory resolves the engine selection to a concrete IBrowserEngine (falling back to Mock when the requested engine is not available in the current build).
  3. The DefaultPolicyEvaluator checks the create + navigation against the BrowserPolicy (origins, bridge channels/types, payload size).
  4. BrowserManager.CreateAsync creates the browser, initializes it, and navigates to the URL.
  5. The platform native adapter (WebView2 / WebGL iframe / WKWebView / …) renders the page; the managed IBrowserInstance exposes navigate / evaluate / dispose.
ⓘ
Same path, two entry points. The component calls exactly the runtime types you would use from code — the no-code layer adds Inspector wiring and a factory, nothing more.

🔗 Bridge & Policy

Talk to the page safely, behind an allow-list policy.

The BridgeBus carries typed request/response and chunked streaming messages between Unity and the page. Every message is evaluated by the DefaultPolicyEvaluator against the BrowserPolicy:

  • Allowed origins — navigation and bridge origins must match (scheme/host/port, or a scheme prefix like astra:).
  • Allowed bridge channels / types — an allow-list (* permits all).
  • Max payload bytes — oversized messages are rejected.
  • Local HTTP — opt-in for http://localhost / 127.0.0.1 during development.
var policy = BrowserPolicy.AllowLocalDevelopment();
policy.AllowedOrigins.Add("https://example.com");
var options = new BrowserCreateOptions { InitialUrl = "https://example.com/", Policy = policy };C#

👤 Profiles

Isolate cookies and storage per profile.

A Browser Profile asset (or a ProfileId in code) keys cookie and storage state. Assign one to the Astra Web View component's Profile Asset slot, or create profiles in code through AstraWebRuntime.Profiles. Persistence modes range from in-memory to file-backed.

📊 Diagnostics & Automation

See what happened, and replay sequences.

  • Diagnostics — every browser/bridge/policy event is recorded to a diagnostics sink. Open Tools → Astra Web Platform → Diagnostics, or export a support bundle.
  • Automation — the automation runner executes a sequence of create/navigate/evaluate/bridge commands; author them as an Automation Script asset.
  • Visual Scripting — browser and workflow units are provided for Unity Visual Scripting.

📙 Scripting API

The code-first path — unchanged and fully supported.

// No-code component, driven from code (auto-open defers to you):
var view = gameObject.GetComponent<AstraWebViewComponent>();
view.SuppressAutoOpen();
await view.OpenAsync("https://example.com/");

// Or the raw runtime:
var engine = AstraBrowserEngineFactory.ResolveEngine(
    AstraBrowserEngineSelection.Automatic, out var id, out _, out _);
var manager = new BrowserManager(engine, AstraWebRuntime.Diagnostics,
    AstraWebRuntime.Profiles, new DefaultPolicyEvaluator(AstraWebRuntime.Diagnostics));
var browser = await manager.CreateAsync(new BrowserCreateOptions { InitialUrl = "https://example.com/" });
string title = await browser.EvaluateScriptAsync("document.title", default);C#

AstraWebViewComponentNo-code MonoBehaviour: OpenAsync, RebuildAsync, CloseAsync, SuppressAutoOpen, Browser, ResolvedEngineId.
AstraBrowserEngineFactoryResolveEngine / EngineIdFor / IsAvailable for the engine enum.

💡 Troubleshooting

The common bumps and their fixes.

The page does not open / "Navigation origin is not allowed"

The URL's origin is not permitted by the policy. With no Settings asset, the component auto-allows the exact URL you typed; if you assigned a Settings asset, add the origin to its Allowed Origins (or use the Allow Local Development preset for testing).

The engine shows "astra.mock" in the Editor

Expected. Native engines are gated to player builds. Build for the target platform (or use WebGL/Windows) to see the real engine. The Inspector's engine summary tells you what will be used.

Windows player shows nothing

Install the WebView2 Evergreen runtime (preinstalled on Windows 11; otherwise download it from Microsoft). Confirm AstraWebView2.dll is packaged under <Player>_Data/Plugins/x86_64/.

macOS / iOS / Linux / Android do not render

These adapters require their platform binaries. Build the native plugin per the cross-platform build guide and validate on the device; until then those engines resolve to Mock.

Console warnings about native toolchain

Project validation reports missing CMake / cl.exe / WebView2 SDK as warnings — they only matter when you build desktop native plugins yourself and do not affect using the prebuilt WebGL/Windows adapters.

❓ FAQ

Do I need to write any code?

No. Add the Astra Web View component (or use the GameObject menu), set a URL, pick an engine, press Play. Code is fully supported but optional.

Will the no-code component override my code?

No. The auto-open is gated: it defers when you call SuppressAutoOpen()/OpenAsync() or turn off Open On Start.

Which platforms render real pages today?

WebGL and Windows are independently validated. The other adapters are build-ready scaffolding to be validated on their hardware.

Can I evaluate JavaScript / call into the page?

Yes — IBrowserInstance.EvaluateScriptAsync, plus the typed bridge for request/response and streaming messages.

Does removing the component change my project?

No. It is additive; nothing is written to disk on your behalf.
