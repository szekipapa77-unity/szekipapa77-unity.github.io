Lucent UI Unity 6000.0.62f1+

One uGUI package — every shape, chart, glow and frosted panel is a generator on a single backend-neutral render spine.

Overview

Lucent UI is the single Unity uGUI package a team installs instead of stitching together five-to-eight single-purpose UI assets. It is a procedural + dynamic suite: every visual — a rounded rect, a chart series, a glow ring, a frosted glass panel — is a geometry generator that writes into one backend-neutral display list, and that one list is drawn by a pluggable surface. Eight buyer-recognizable niches ride that same spine.

  • N1 Procedural shapes — analytic rounded rects, ellipses, polygons, arcs, dashed/dotted strokes.
  • N2 Soft shadow / glow / outline / neumorphism — geometry effects composed onto the same element.
  • N3 Blur / translucency / liquid glass — GPU frosted panels and refractive glass.
  • N4 Animated buttons + a self-contained UI motion engine.
  • N5 Charts / data-viz — line, area, pie, scatter, all tessellated through the spine.
  • N6 Scroll / recycler — virtualized lists and grids at zero steady-state allocation.
  • N7 Localization / RTL / animated text — real Unicode bidi, Arabic shaping, per-glyph effects.
  • N8 Theming / widgets / responsive layout — design-token theming and the framework backbone.

One spine, many surfaces

The whole asset is built on the Lucent.Render spine. A generator implements ILcGenerator and writes pre-tessellated triangles into an LcDrawList (a list of LcVertex + indices + non-tessellated LcTextRun labels). A backend implements exactly one interface — ILcRenderSurface — to turn that list into a native mesh. Because tessellation happens once in the list, every backend renders identical pixels:

// A generator writes backend-neutral geometry...
var list = new LcDrawList();
var ctx  = new LcDrawContext(rect, list);
generator.Generate(ctx);             // ILcGenerator -> LcDrawList

// ...and ANY surface turns that one list into one native mesh.
ILcRenderSurface surface = GetComponent<LcUGuiSurface>();
surface.Submit(list);                // uGUI overlay, UI-Toolkit, or world-space

The primary surface, LcUGuiSurface, is a MaskableGraphic that maps the list to a CanvasRenderer mesh in OnPopulateMesh. LcUIToolkitSurface and LcWorldSpaceSurface consume the very same LcDrawList. The authoring base every component extends, LcDrawable, runs the generator and its sibling effect chain in painter order into one pooled list — so a shape plus its stroke, shadow, glow and gradient collapse into a single mesh.

One mesh = one draw call. A shape and all of its geometry effects emit into a single LcDrawList in LcDrawLayer order (Shadow → Fill → Stroke → Glow → Overlay), so list-order equals painter-order with no per-effect renderers. Identically-configured elements share one cached material and batch. (Pooled child Text overlays are the only intrinsic extra draw.)

Why Lucent — the headline wins

  • Analytic anti-aliasing, crisp at any scale. Each LcVertex carries a signed edge distance and a local AA half-width in Uv1; the shared LcCanvas shader smoothsteps the edge, so curves stay sharp at arbitrary zoom — no MSAA, no oversized textures, no mipped sprite atlas.
  • 0-GC scrolling. LcRecycler keeps only visible + 2×buffer cells alive regardless of item count; the steady-state scroll hot path allocates zero bytes (struct enumerators, reused scratch buffers, no LINQ/closures/boxing), so LcScrollView scrolls hundreds of thousands of rows flat.
  • GPU liquid glass, one component. LcGlass is an LcBlurPanel plus analytic refraction, chromatic aberration, an edge glint, squircle corners and a native soft shadow — all in one mesh and one shared material. A three-asset stack (translucent blur + glass + shadow) collapses into a single batched element.
  • Real RTL / bidi. LcTextRtl.Convert and the Lucent.Text.Unicode.BiDi pipeline implement UAX #9 reordering with Arabic contextual shaping and lam-alef ligatures — not a naive "RTL fixer."
  • No TextMeshPro required. Text renders through the built-in LegacyRuntime.ttf overlay font (ships with every Unity install — zero import step) or the self-contained LcSdfText distance-field renderer, which shapes raw .ttf bytes with their own OpenType tables — no TMP, no native code.
  • ~99% coverage of the competitor superset. The V1 ship set is a deliberate superset of the union of every competing asset's features (the Competitive-Coverage Law). The audited number is 215/217 = 99.1%; the only two residuals are external (the official Unicode bidi conformance corpus and binary showcase sprites — the asset is procedural by design).

Lucent UI 0.1.0 targets Unity 6000.0+ and has no external package dependencies and no setup step. Where the market forces a buyer to assemble a roughly EUR 290-365 stack of 7-8 SKUs with clashing conventions and batching that breaks at the seams, Lucent is one purchase, one spine, one theme system, and guaranteed batching.

Installation & Requirements

Lucent UI is a uGUI-native asset: it draws through Unity's existing CanvasRenderer spine, so it slots into any project that already has a Canvas. There is no external SDK, no native plugin, and no service to register. Drop the Assets/Lucent folder in and the runtime self-initializes.

Requirements

  • Unity 6000.0.62f1 or newer. The project is authored and validated against 6000.0.62f1; the runtime uses the modern FindAnyObjectByType path under a UNITY_2023_1_OR_NEWER guard, so it stays correct on the whole Unity 6 line.
  • com.unity.ugui (2.0.0+) — required. This is the only hard dependency. TextMeshPro is merged into this package (it ships Unity.TextMeshPro.asmdef under com.unity.ugui/Runtime/TMP), so there is no separate TMP import or "Import TMP Essentials" step gating Lucent text.
  • New Input System (com.unity.inputsystem) — recommended, not required. Lucent is input-backend-optional. Its asmdefs raise the LUCENT_INPUTSYSTEM versionDefine only when the package is present, and the interaction bridge (charts pan/zoom, LcScrollView) prefers UnityEngine.InputSystem when that symbol is defined, falls back to the legacy Input manager otherwise, and finally to a zeroed no-op so it still compiles and runs with no input backend at all.
  • URP (com.unity.render-pipelines.universal 17.0.0+) — optional, enables the blur/glass hero. The core effects assembly takes no URP reference; the URP scene-color grab lives in a separate Lucent.Effects.Urp assembly gated by the LUCENT_URP versionDefine + define-constraint, so it only compiles when URP is installed.
  • HDRP (com.unity.render-pipelines.high-definition 17.0.0+) — optional. Same pattern: Lucent.Effects.Hdrp is constrained to LUCENT_HDRP and provides the custom-pass blur path. No HDRP, no compile cost.

Why the optional backends never break the build: each integration sits behind a versionDefine declared in its .asmdef. The packages do not need to be present for the asset to compile — the gated code simply drops out. The same mechanism backs UI Toolkit (LUCENT_HAS_UITOOLKIT, raised from com.unity.modules.uielements).

// Packages/manifest.json -- only com.unity.ugui is mandatory; the rest light up features.
"dependencies": {
  "com.unity.ugui": "2.0.0",
  "com.unity.inputsystem": "1.14.2",                  // recommended -> LUCENT_INPUTSYSTEM
  "com.unity.render-pipelines.universal": "17.0.4"    // optional   -> LUCENT_URP (blur/glass)
}
// The versionDefine seam, e.g. Lucent.Effects.Urp.asmdef -- present only when URP is installed:
"defineConstraints": [ "LUCENT_URP" ],
"versionDefines": [
  { "name": "com.unity.render-pipelines.universal", "expression": "17.0.0", "define": "LUCENT_URP" }
]

Importing

  1. Import the package (or copy Assets/Lucent into your project). Unity compiles the Lucent.* assemblies; the optional URP/HDRP/Input bridges resolve automatically based on which packages your manifest already has.
  2. Set Active Input Handling. In Edit > Project Settings > Player > Active Input Handling, choose Input System Package (New) (recommended) or Both. This project ships with the New System enabled (activeInputHandler: 1).
  3. No further setup. The composition root LcSdkBootstrap runs at RuntimeInitializeOnLoadMethod(BeforeSceneLoad) and builds the default LcSdk.Registry (every shape, effect, chart, scale, palette) once at startup. It is batch-guarded: under -batchmode it builds the registry but does no UI side effects, so headless/CI runs never hang.

Your first scene

Lucent components are ordinary uGUI Graphics, so the host is a normal Canvas. Create one via GameObject > UI > Canvas, add a child UI object, then add a Lucent component — from Add Component > Lucent UI > Lc Shape, or in code:

using Lucent.Core;
using Lucent.Shapes;
using UnityEngine;

// uiChild is any RectTransform under a Canvas.
// A freshly added LcShape already renders a correct AA rounded rect with zero config.
var shape = uiChild.AddComponent<LcShape>();
shape.ShapeKind = LcShapeKind.RoundedRect;
shape.SetUniformRadius(24f);
shape.SetFillColor(new Color32(38, 132, 255, 255));   // one mesh, one draw call

To see the URP/HDRP hero, add the flagship glass panel. It always renders; when no live scene-color grab exists on the device/pipeline it degrades gracefully to a flat tinted translucent fill rather than a broken shader:

using Lucent.Effects;

var glass = uiChild.AddComponent<LcGlass>();   // frosted blur + refraction + chromatic aberration + edge glint
if (!glass.IsSupported)
{
    // LcCapabilities.SupportsSceneColorGrab == false (e.g. Built-in with no grab provider):
    // the panel falls back to a flat tinted fill. No pink shader, ever (invariant 7).
}

Pipeline is detected at runtime, not configured. LcCapabilities.ActivePipeline reads GraphicsSettings.defaultRenderPipeline and returns LcPipeline.BuiltIn, Urp, or Hdrp; the auto-created LcBlurManager re-detects every frame, so a runtime pipeline swap is handled without any per-pipeline asset variant.

  • One mandatory dependency (com.unity.ugui) — everything else is opt-in and gated behind versionDefines, so missing packages drop out instead of breaking compilation.
  • TextMeshPro needs no separate install: it is merged into com.unity.ugui.
  • Input-backend-optional: New Input System, legacy Input manager, or neither — the same code path compiles and runs in all three via LUCENT_INPUTSYSTEM.
  • Cross-pipeline by runtime detection (LcCapabilities.ActivePipeline): Built-in, URP, and HDRP from one asset, with the glass/blur hero degrading to a flat tint where no scene-color grab is available.
  • Zero-setup startup: LcSdkBootstrap self-initializes the registry and is batch-guarded for headless CI.

Procedural Shapes

LcShape is the no-code authoring component for vector shapes. It is an LcDrawable — a MaskableGraphic on Lucent's shared DrawList spine — so a single shape composes its fill, gradient/pattern/sprite paint, and stroke into one mesh, one draw call. A freshly added component renders a correct rounded rect on Play with zero setup: the serialized LcShapeStyle block ships sensible defaults and is the source of truth. Code setters write that same block and mark the mesh dirty, so an Animator, Timeline, or script always wins over the inspector, and the build path is fully deterministic (no Random/DateTime/statics).

The shape family is the LcShapeKind enum — thirteen generators, each a registry row behind LcShapeGenerators.Create:

public enum LcShapeKind
{
    RoundedRect, Circle, Ellipse, Polygon, Star, Chamfer,
    Parallelogram, Path, Arc, Pie, Ring, Capsule, Squircle
}

Setting ShapeKind swaps the cached generator (LcShapeGeneratorBase); the live LcShapeStyle instance is shared straight through, so a kind change costs no per-frame allocation. Each generator implements only its silhouette (EmitFill) and, where it differs, its stroke; the base class runs Fill then Stroke layers in order through the spine's LcGeometryBuilder primitives.

Per-corner radius

Corners are an LcCorners struct ordered TL, TR, BR, BL (pixels). A shape can use one UniformRadius for all four or independent PerCornerRadius values; radii are Clamped to half the shorter side so they never invert. Capsule is just a rounded rect whose radius is locked to half the shorter side, and Squircle swaps the circular corner for a superellipse profile (SuperellipseExponent ~= 5 gives the iOS look).

var shape = go.AddComponent<LcShape>();
shape.ShapeKind = LcShapeKind.RoundedRect;          // rebuilds the generator
shape.SetCornerRadii(new LcCorners(16, 16, 4, 4));  // TL, TR, BR, BL
shape.SetFillColor(new Color32(38, 132, 255, 255));

// Stroke fields, then one dirty-marking setter picks them all up:
shape.Style.StrokeAlign = LcStrokeAlign.Inner;      // Inner / Center / Outer
shape.Style.Dashed = true;                          // marching-ants capable
shape.SetStroke(enabled: true, width: 3f);
shape.SetStrokeGradient(true, Color.white, Color.cyan, angleDeg: 45f);

Fills

LcFillKind selects how the interior is painted, and the shared spine keeps every mode on the always-batches path:

  • Solid — a flat FillColor, or a theme color token (FillTokenId) resolved through the bound LcTheme when one is present (code defers to the literal otherwise).
  • Gradient — vertex-colored over the fill range via LcGeometryBuilder.ApplyGradient. Families are Linear, Radial, Angular, Stepped, and Corner4; a theme GradientTokenId can override the literal stops (1.18).
  • Sprite — the shape silhouette clips a sprite. LcShape.mainTexture binds the sprite's texture so the shared white-sampling material samples it, with uGUI Image parity draw modes (Simple, Sliced, Tiled, Filled + fill method/amount).
  • Pattern — a procedural Grid/Checker/Stripe (LcPatternKind). GetModifiedMaterial hands back a per-instance material copy carrying the SHADER agent's pattern uniforms, so each shape patterns independently while the shared material is never mutated.

Strokes

The outline is emitted into the Stroke layer of the same mesh. Width is per-side: the four side toggles collapse into an LcEdgeWidths (Top/Right/Bottom/Left), and any side at width 0 is simply omitted by StrokeRectPerSide — so "underline only" or "no top border" needs no extra objects. LcStrokeAlign places the centerline Inner, Center, or Outer of the silhouette. Strokes can be solid, dashed (DashLength/DashGap, animatable for marching ants), or dotted (zero dash length routes to DottedPolyline), and the stroke layer can carry its own independent gradient, separate from the fill.

Analytic anti-aliasing

AA is built into the geometry, not bolted on with MSAA or a per-shape texture. Every fill emits a solid interior inset by the AA half-width plus a feather ring offset outward along the edge normal and faded to alpha 0. Each vertex carries an analytic distance field in LcVertex.Uv1 (x = signed distance to the edge in px, y = local AA half-width) that the shared LcCanvas shader turns into a smoothstep — so edges stay crisp at any scale or DPI:

// LcGeometryBuilder.EmitFilledOutline (abridged)
float hw = aa <= 0f ? 0f : aa * 0.5f;                 // AA half-width, px
Color32 outerColor = new Color32(color.r, color.g, color.b, 0);

// inner ring: full color, positive signed distance
dl.AddVertex(new LcVertex(ring[i] - nrm[i] * hw, color, CenterUv,
                          new Vector2(coreDist, hw)));
// outer ring: alpha 0, negative signed distance -> shader feathers the edge
dl.AddVertex(new LcVertex(ring[i] + nrm[i] * hw, outerColor, CenterUv,
                          new Vector2(-hw, hw)));

Concave outlines (an authored Path, a deep star notch) would self-overlap under a centroid fan, so FillPolygon detects them and ear-clips through LcPolygonTriangulator, then feathers along the true edge normals. Convex and star-convex inputs keep the fast centroid-fan path. Arc/Pie/Ring fills route through WedgeAA for the same feathered edge.

Rounding-aware raycast

LcShape implements ICanvasRaycastFilter, so a click is valid only when it lands inside the shape's actual silhouette. The same perimeter that drives the mesh is published through ILcSilhouetteSource.TryGetSilhouette (also consumed by shadow/glow/outline effects so they trace the real outline) and tested with an even-odd LcShapeRaycast.PointInPolygon that handles convex and concave rings alike. A press in a rounded-off, star-notch, or chamfered corner therefore misses the graphic and reaches whatever is behind it. Open/annular families (Arc, Pie, Ring) have no simple closed silhouette and fall back to rect containment, preserving normal clickability.

  • Thirteen shape families from one component, correct on Play with zero configuration.
  • Fill + gradient/pattern/sprite + per-side stroke batch into one mesh = one draw call.
  • Analytic, geometry-side AA stays crisp at any scale — no MSAA, no per-shape render texture.
  • Clicks honor the true outline: rounded, star, and chamfered corners correctly miss.
  • Steady-state rebuilds are zero-GC — pooled stroke/fill gradients, silhouette buffer, and pattern material.

Effects & Liquid Glass

Lucent ships two tiers of decoration. N2 is the batched geometry effect set -- soft drop shadow, true outer glow, and a crisp vector outline -- emitted as real triangles into the host mesh. N3 is the scene-sampling hero: a frosted background-blur panel and the unified liquid glass element, driven by one dual-Kawase blur source that runs on URP, the Built-in pipeline, and HDRP from a single piece of math.

N2: shadow, glow, outline as no-code components

Each effect is authored by a thin MonoBehaviour over a serialized pure-logic spec. LcShadow, LcGlow, and LcOutline all derive from LcEffectBehaviour, which implements the L1 ILcEffect interface and delegates Apply(LcEffectContext) to its spec (LcDropShadowEffect / LcGlowEffect / LcOutlineEffect). A sibling LcDrawable discovers them by GetComponents and folds their geometry into the same mesh as the shape -- so a shadow plus a glow plus an outline on one element is still one draw call, no RenderTexture per element. Add them and they just work: the shadow defaults to offset (0,-4), blur 8, 35% black; the glow to cyan at radius 12; the outline to 2px white, center-aligned.

The falloff is not a vertex-color approximation. LcEffectGeometry emits a feathered ring whose outer vertices are alpha-0 and carry a negative Uv1 signed distance, so the soft edge runs through the same analytic-AA model as the shape fills: a wide smoothstep under the LcCanvas distance shader, and the alpha ramp itself under a plain vertex-color fallback.

Casting effects on stock uGUI: LcGraphicEffect

The True-Shadow parity feature is LcGraphicEffect: it renders a shadow / glow / outline behind stock uGUI graphics Lucent does not own -- Image, RawImage, Text, TMP. It derives the host silhouette from the graphic's rect (and, for an Image carrying an authored physics shape, its alpha outline via ILcSilhouetteSource.TryGetSilhouette), runs the same LcEffectGeometry emit path, and presents the result through a hidden sibling LcGraphicEffectRenderer ordered just before the host so it paints underneath. Every Normal-blend effect rides that one surface -- one mesh, one extra draw call, RenderTexture-free -- and it is deterministically rebuildable headlessly via RebuildEffectNow().

// 1) No-code on any LcDrawable (e.g. an LcShape): composed INTO the shape's own mesh.
var shadow = shape.gameObject.AddComponent<LcShadow>();   // (0,-4), blur 8, 35% black
shape.gameObject.AddComponent<LcGlow>();                  // cyan, radius 12
shape.gameObject.AddComponent<LcOutline>();               // 2px white, Center

// 2) Cast the SAME effects behind a STOCK uGUI Graphic Lucent does not own.
var fx = image.gameObject.AddComponent<LcGraphicEffect>();
fx.ShadowEnabled = true;
fx.Glow.Color = new Color32(64, 224, 255, 255);
fx.Glow.Blend = LcBlendMode.Additive;     // additive bloom -> its own extra draw
int verts = fx.RebuildEffectNow();        // deterministic, headless-rebuildable

Blend modes without breaking the batch

Each effect pass carries a per-effect LcBlendMode (Normal, Additive, Screen, Multiply). Normal is byte-identical to the shared LcCanvas mesh blend, so it stays inside the single batched draw. The other three cannot recolour that shared straight-alpha mesh, so each distinct non-Normal mode in use gets exactly one extra surface (LcEffectBlendSurface) whose material carries that mode's fixed-function blend state. The mapping lives in LcEffectBlend.StateFor:

case LcBlendMode.Additive:  // out = src.rgb*src.a + dst : adds light
    return new LcBlendState(BlendMode.SrcAlpha,        BlendMode.One,               BlendOp.Add);
case LcBlendMode.Screen:    // 1-(1-s)(1-d) : soft highlight, never clips dark
    return new LcBlendState(BlendMode.OneMinusDstColor, BlendMode.One,              BlendOp.Add);
case LcBlendMode.Multiply:  // alpha-aware darken
    return new LcBlendState(BlendMode.DstColor,        BlendMode.OneMinusSrcAlpha,  BlendOp.Add);

The Lucent/LcCanvas shader bakes Blend SrcAlpha OneMinusSrcAlpha into every SubShader, so writing _SrcBlend/_DstBlend onto a clone of it is a dead no-op. LcEffectMaterial therefore builds each variant from a blend-driven pass -- the shipped Lucent/LcEffectBlend shader, degrading to Hidden/Internal-Colored, then to the canvas material -- so a stripped shader renders Normal rather than going magenta. An author can also assign a per-pass CustomMaterial (e.g. an animated noise glow), resolved verbatim by LcEffectMaterial.Resolve.

For canvas-wide batching, LcEffectLayer collects many elements' effects (each via ILcLayerEffect.EmitLayerEffects in its own local rect space), transforms them into the layer's space, and emits them into one pooled LcDrawList -- one mesh, one draw -- instead of True Shadow's RenderTexture-per-element.

N3: the blur / liquid-glass hero

LcBlurPanel is a no-code frosted/acrylic panel: it emits one Overlay rounded-rect or squircle quad whose material (the LcBlur shader) samples the global _LucentBlurTex at the panel's screen-space UV, with tint, vibrancy (saturation), brightness, grain, and a per-panel blur Radius that drives the shader tap spread. LcGlass is an LcBlurPanel plus analytic refraction, chromatic aberration, an edge glint, and a native under-glass soft shadow (the geometry-shadow path, same mesh) -- so Paraform's three-asset stack (Translucent Image + Paraform + True Shadow) collapses into one component. The shadow is emitted first (behind), then the glass quad, both in one list, so painter order gives the layered look with no separate draw.

var glass = panel.AddComponent<LcGlass>();    // sensible just-works defaults on add
LcBlurPanelSpec s = glass.Spec;               // getter returns the struct by value
s.Radius  = 24f;                              // BLR-14: per-panel blur radius
s.Quality = LcBlurQuality.High;               // requests a shared quality from the manager
glass.Spec = s;                               // setter re-applies material + rebuilds the quad

glass.GlassSpec = new LcGlassSpec {           // the liquid-glass half
    Refraction = 0.02f, ChromaticAberration = 0.004f, EdgeGlint = 0.5f, NativeShadow = true
};
if (!glass.IsSupported) { /* degrades to a flat tinted fill -- never magenta */ }

Panels whose specs are equal share one pooled material via LcPanelMaterialKey + LcMaterialCache and batch together; LcGlass pools by the full glass key (blur params plus refraction/aberration/glint). When no live scene grab exists (LcCapabilities.SupportsSceneColorGrab == false) the panel still renders -- as a flat tinted translucent fill -- so it never magentas and never null-refs.

One blur, three pipelines

LcBlurManager is the auto-created, one-per-scene coordinator. It detects the active pipeline and publishes the shared LcBlurProfile + LcBlurQuality down to the L1 LcBlurRequest -- the backend-neutral seam that lets the L4 panels and the L4 URP feature both reference it downward, never each other. It also owns the active-panel ref-count, so the grab+blur runs only while a panel is alive, and bumps a resize epoch so the buffers stay resolution-independent.

The profile is just three knobs (downsample factor, iterations, radius), tuned per quality preset by LcBlurProfile.ForQuality:

QualityDownsampleIterationsRadius
Low4 (quarter-res)28
Medium2 (half-res)416
High2524
Ultra2740

Every backend publishes the same _LucentBlurTex from the same dual-Kawase down/up pyramid, just through a different recording API:

  • URP -- LcBlurFeature / LcBlurPass: a RenderGraph ScriptableRendererFeature (the only runtime asmdef that touches URP). It grabs cameraOpaqueTexture (or the UI-inclusive activeColorTexture when a panel requests blur-behind-UI), runs a Blitter down/up pyramid, with an optional compute fast path guarded by SystemInfo.supportsComputeShaders. Gated on LcBlurRequest.AnyActive, so a present-but-idle feature costs nothing.
  • Built-in -- LcBlurBuiltInManager + LcBlurCommandBuffer: no SRP package needed (only always-present CommandBuffer/Camera APIs). The manager auto-attaches a command buffer at CameraEvent.BeforeImageEffects while panels are active and flips the capability flag so glass binds the grab instead of degrading. The pyramid is recorded by PopulateBlurCommandBuffer -- the byte-for-byte equal of the URP path, headlessly testable down to its exact blit count.
  • HDRP -- LcBlurCustomPass: a CustomPass on a volume that ping-pongs between two half-res buffers (no undefined read==write self-blit) and publishes the final index. The whole assembly is gated by LUCENT_HDRP so it compiles out cleanly where HDRP is absent.
  • On-mesh shadow + glow + outline compose into the host LcDrawable mesh: zero extra draws, no RenderTexture-per-element.
  • LcGraphicEffect gives stock Image/Text/TMP the True-Shadow look in one extra draw for all Normal-blend effects; at most one more per distinct non-Normal blend mode.
  • LcGlass unifies Translucent Image + Paraform + True Shadow into a single component and a single shared material; equal-spec panels batch.
  • One dual-Kawase implementation; three pipeline backends publish the identical _LucentBlurTex.
  • Blur work is ref-count-gated -- it runs only while a panel is alive -- and degrades to a flat tint where no scene grab exists (never magenta, never a null-ref).

Animated Buttons

The N4 button, LcButton, is a procedural, fully animated control that extends UnityEngine.UI.Selectable. Because it subclasses Selectable, keyboard and gamepad navigation, the Interactable flag, and the Normal / Highlighted / Pressed / Selected / Disabled state machine all come for free and work on any input backend through uGUI's EventSystem — there is no hard reference to Unity.InputSystem. Instead of Selectable's built-in ColorTint, the button drives a stack of animation channels on every state transition, layers per-event audio over the top, and pairs cleanly with an LcShape host graphic for a one-draw-call procedural skin.

A freshly added LcButton just works: with an empty channel list it applies a built-in Color + Scale default at runtime, so it reacts to hover, press, and click with zero setup. Tweening runs on Lucent's own deterministic LcEasing motion engine — there is no DOTween dependency.

The channel stack

Each entry in the stack is an LcButtonChannel: one LcChannelKind driving one target, holding one value per visual state. Multiple channels run in parallel per transition (multi-target, multi-transition). Ten channel kinds are supported: the six continuous kinds that tween over a Duration with an Ease or AnimationCurve (Color, Scale, Move, Rotate, Fill, Fade), and the four discrete kinds that apply instantly (SpriteSwap, Animator, Activate, Particle). Per-state values live in tidy five-slot structs (LcStateColor, LcStateVector3, LcStateFloat, and friends), each with a Get(state) accessor and convenient DefaultTint() / DefaultScale() factories.

var btn = gameObject.AddComponent<LcButton>();

// One channel = one property, with a value per visual state.
btn.Channels.Add(new LcButtonChannel
{
    Kind            = LcChannelKind.Scale,
    Scales          = LcStateVector3.DefaultScale(), // grow on hover, sink on press
    Ease            = LcEase.OutBack,
    Duration        = 0.16f,
    VariationAmount = 0.25f                           // per-press feel jitter
});
btn.Channels.Add(new LcButtonChannel
{
    Kind     = LcChannelKind.Color,
    Colors   = LcStateColor.DefaultTint(),
    Ease     = LcEase.OutQuad,
    Duration = 0.12f
});
btn.Refresh();   // re-bind targets to the host graphic/rect and re-apply the current state

Empty target slots fall back to the button's host graphic (HostGraphic) and host rect (HostRect), so a channel "just binds" without wiring. On Bind each channel captures its untouched home pose once, so Move/Rotate/Scale values stay relative to your layout, and every continuous tween starts from the current live value — a code-driven value is never stomped mid-flight. A non-Once Loop turns a channel into an idle pulse (oscillating Normal<->Highlighted) that runs while LoopIdle is on and the button is in the Normal state.

Presets

The LcButtonPresets factory ships 35 built-in stacks enumerated by LcButtonPresetKind — from HoverScale, PressPunch, and Jelly through Heartbeat, GlowPulse, RubberBand, and ShimmerTint. Each is a pure, deterministic method returning a fresh List<LcButtonChannel> with null targets, so it binds to whatever host you apply it to. Presets are reachable three ways: in code via LcButtonPresets.Create(kind), as a runtime ScriptableObject via LcButtonPreset.CreateBuiltin(kind), or as a shareable [CreateAssetMenu] LcButtonPreset asset assigned to LcButton.Preset.

// Apply one of the 35 built-in stacks, then drive the button from code.
btn.Channels.Clear();
btn.Channels.AddRange(LcButtonPresets.Create(LcButtonPresetKind.Jelly));
btn.Refresh();

btn.SetContent(iconSprite, "Play");   // icon sprite + animated label in one call
btn.Randomize();                      // advance the seeded per-press variation
btn.Play(LcTriggerKind.Press);        // fire audio + UnityEvent + drive the Pressed state

Precedence: code always wins. An assigned LcButtonPreset overrides the inline channel stack (a preset is the more specific authored intent, and ApplyTo clones its channels so the shared asset is never mutated by a button's runtime home-capture). But a programmatic Play() or SetState() latches a code-configured flag that prevents the auto default or preset from ever re-stomping your stack.

Audio, ripple, entrance, and label

The serialized LcButtonAudio block holds a clip per trigger (Click / Hover / Press / Select / Deselect / Disabled), a base volume and pitch, and — net-new versus the parity competitors — deterministic pitch/volume jitter plus optional AudioMixerGroup routing. Randomization draws from a seeded LcRng (never UnityEngine.Random), so the variation order is reproducible, and clips play through a pooled AudioSource.PlayOneShot with no per-click allocation. With no clips assigned it is a silent no-op.

A press emits an expanding-circle ripple through LcButtonRipple, which is itself an LcUGuiSurface: the ripple is not a bespoke effect but rides the same shared LcDrawList spine as every other Lucent primitive, so the whole live batch resolves to one mesh / one draw call. It auto-wires when dropped on a child of the button, never steals the button's raycast (raycastTarget = false), and ticks only while ripples are in flight — steady-state zero work. A pointer press ripples at the cursor; a keyboard or gamepad Submit ripples from the button center.

The Start trigger plays a real entrance animation: PlayEntrance() snaps the one-shot continuous channels to a collapsed Pressed pre-roll pose, then animates them into the resting state so the button visibly pops in (without latching the code-config flag, since an intro is automatic, not authored). The optional LcButtonLabel bridge routes the button's text through Lucent.Text's per-glyph LcTextAnimator and maps each state to a per-glyph action (replay the reveal on hover/select, pause when disabled) — inert and harmless when no animator is bound.

Deterministic variation. Randomize() advances a seeded [0,1) Variation value that the button feeds into each channel. A channel's VariationAmount (0..1) then modulates its tween duration and its Move/Rotate/Scale travel-from-home by 1 + (variation*2-1)*VariationAmount — so per-press feel varies, yet stays fully reproducible from the seed. With VariationAmount == 0 it is inert.

  • Free navigation + state machine: subclassing Selectable gives keyboard/gamepad focus, Interactable, and the five-state transition flow on any input backend via EventSystem — no Unity.InputSystem reference.
  • Zero-setup default: an empty button self-applies a Color + Scale stack and reacts to hover/press/click immediately.
  • One draw call ripple: the press ripple rides the shared LcDrawList spine and ticks only while ripples live (steady-state zero work, never blocks the raycast).
  • Reproducible audio: seeded LcRng pitch/volume jitter (never UnityEngine.Random) through a pooled PlayOneShot with optional mixer routing — zero per-click allocation.
  • No DOTween: all tweening runs on Lucent's self-contained LcEasing engine; Tick(dt) is public so EditMode/headless tests can pump animation deterministically.

Charts

The charts layer turns an LcDataset into a picture without ever stepping off the shared render spine. LcChartView is not a wrapper around a separate chart canvas — it is an LcDrawable (which is an LcUGuiSurface). Its bound generator writes the chart's panel, axes, series and legend into one pooled LcDrawList inside OnPopulateMesh, exactly the way LcShape does. The upshot is that an entire chart — gridlines, bars, value labels and all — batches into a single mesh and one draw call, and rides the same render path as every other Lucent visual.

The component works unconfigured: a freshly added LcChartView with no dataset renders a default line chart over a bundled 8-point sample, so nothing on screen is ever blank-by-default. Configuration comes from two directions that never fight: the inspector (a serialized LcChartKind plus X/Y/value column names, axis and legend toggles), and a fluent code builder. A generator assigned in code latches a _generatorFromCode flag so the inspector can never silently override it.

The fluent builder

var sales = new LcDatasetBuilder("sales")
    .Category("month", "Jan", "Feb", "Mar", "Apr")
    .Number("revenue", 12, 19, 7, 22)
    .Build();

LcChartView.Create(canvas.transform)
    .WithData(sales)
    .Bar("month", "revenue")   // or .Line / .Scatter / .Area / .Spline / .Step / .Bubble(...)
    .Themed(theme)             // series/panel/axis colors pulled from theme tokens
    .Axes()                    // bottom X + left Y cartesian axes
    .Legend(LcLegendVisibility.Auto, LcAxisSide.Right)
    .Build();

Beyond the cartesian shortcuts, .Chart(LcChartGeneratorBase) binds any generator in the catalog. Composition is deterministic and order-driven: Generate emits the themed panel first (so it lands behind everything), reserves axis thickness and a legend strip through the box-model LcChartLayout.Solve(), projects the generator into the shrunk plot rect, then draws the legend keys into the space the plot was carved out of — no overlap by construction.

The generator catalog

Every chart type is an LcChartGeneratorBase implementing the spine's ILcChartGenerator. The base class owns column binding (Dataset, XColumn, YColumn), color and stroke, NaN-tolerant projection (ProjectFinite drops any non-finite point so no NaN geometry is ever emitted), animated reveal via LcDrawContext.AnimationT, and per-row hit-test recording (LastPoints / LastRows). Each generator reports a stable TypeId ("line", "bar", "candlestick", ...) surfaced as LcTypeId ("lc.chart.line") for the registry.

  • Line family — LcLineGenerator, LcAreaGenerator (fill to baseline + outline), LcSplineGenerator (Catmull-Rom), LcStepGenerator, LcSparklineGenerator.
  • Point family — LcScatterGenerator (per-row markers), LcBubbleGenerator (a third column drives marker radius).
  • Bar family — LcBarGenerator, LcHorizontalBarGenerator, LcStackedBarGenerator, LcGroupedBarGenerator (the multi-value bars tint each series from LcColorScale.Sequential()).
  • Radial — LcPieGenerator, LcDonutGenerator, LcRadarGenerator (concentric grid rings + spokes), LcSunburstGenerator.
  • Statistical — LcHistogramGenerator, LcBoxPlotGenerator, LcViolinGenerator (mirrored density outline), LcHeatmapGenerator (row-major grid colored by a sequential scale), LcGaugeGenerator, LcProgressGenerator.
  • Relational / flow — LcTreemapGenerator (slice-and-dice), LcSankeyGenerator (curved smoothstep ribbons), LcParallelCoordinatesGenerator, LcTimelineGenerator, LcGanttGenerator, LcWaterfallGenerator, LcFunnelGenerator.
  • Composite — LcMultiAxisGenerator (each series on its own auto Y scale over a shared X) and LcComboGenerator, which forwards the shared context and dataset to a list of heterogeneous child generators (e.g. bars under a line) and merges their hit points so pointer interaction spans the whole composition.

The breadth set beyond the core thirteen kinds (histogram through combo) is appended at the end of LcChartKind so existing inspector ordinals stay stable. Box and violin compute quartiles with a local deterministic type-7 percentile over a sorted copy — identical input always yields identical geometry, which is what makes the charts headless-verifiable.

Scales, axes, legend, tooltips

Positional scales implement ILcScale (double-precision Project/Invert/Ticks): LcLinearScale, LcLogScale, LcSymlogScale (linear near zero, log beyond — handles negatives), LcPowerScale, LcTimeScale (Unix-seconds domain with calendar-aware nice ticks), plus the discrete LcOrdinalScale and LcBandScale that the bar generators use for evenly-padded category bands. Tick text comes from an ILcTickFormatter — LcNumberFormatter, LcSiPrefixFormatter, LcPercentFormatter, LcCurrencyFormatter, LcDateTimeFormatter, or a delegate. Color mapping is kept separate in LcColorScale (sequential / diverging multi-stop gradients) because it returns a color rather than a position.

LcChartLayout is a pure box-model solver: from an outer rect it subtracts padding, an optional title band, the legend strip and each axis's reserved thickness to compute the plot rect, then assigns the resulting pixel range back onto each axis's scale. LcAxisRenderer draws baselines, ticks, gridlines, tick labels and titles into the Stroke layer; LcLegendRenderer stacks (vertical sides) or flows (horizontal sides) one swatch + LcTextRun per LcLegendEntry, clipped to its reserved slot. Both emit into the same LcDrawList, so axes and legend batch into the chart's one mesh.

Interaction is device-agnostic and headless-friendly. LcHitTester.NearestIndex does the spatial query against the generator's projected points; LcChartView.OnPointerClick and an optional per-frame hover poll drive an LcSelectionModel and an LcTooltipController. LcPanZoomController mutates scale domains (not pixels) to window the data, clamped to a cumulative zoom range. The LcInputAdapter bridge prefers the new Input System and falls back to the legacy manager (or a zeroed no-op), so the stack imports and runs on any input backend.

World-space and UI Toolkit surfaces

Because the chart's output is a backend-neutral LcDrawList, the same chart can target any surface that implements ILcRenderSurface (PlotRect + Submit). LcChartView.RenderToSurface composes the full chart against the surface's plot rect and hands it over — without Lucent.Charts ever naming a concrete surface assembly. LcWorldSpaceSurface maps the list onto a planar Mesh (a flat quad in the 3D scene for billboards or diegetic HUDs) and is the recommended large-mesh target because it promotes to a 32-bit index buffer above 65535 vertices. LcUIToolkitSurface is a VisualElement that emits the list into its repaint mesh via MeshGenerationContext, so a chart can live inside a UXML/USS hierarchy beside native UITK controls (16-bit indices; y flipped to top-left).

Headless export

// Compose the chart onto the headless capture surface, then export -- no GPU.

chart.RenderToSurface(capture);                  // panel + axes + series + legend

LcRasterExporter is a CPU rasterizer: it barycentric-fills the draw list's triangles with per-vertex color interpolation and alpha blend, blits text runs through the built-in LcBitmapFont, and encodes to PNG/JPG via Unity's deterministic image encoder. LcVectorExporter takes the true vector path — each triangle becomes an SVG <polygon>, each label a <text> — with no rasterization and no GUIDs/timestamps. LcCsvExporter and LcJsonExporter round-trip the underlying LcDataset (RFC-4180 CSV / flat JSON rows, invariant culture, BOM-less). LcReportComposer merges many chart panels into a titled near-square grid, paginates at a fixed capacity, and flows each page straight into the raster or vector exporter.

Dashboard

LcDashboardView is a MonoBehaviour that hosts LcChartView tiles and only drives their RectTransforms — each child still batches into its own single mesh. N6 - Virtualized Scroll: the 0-GC recycler

A uGUI ScrollRect instantiates one GameObject per row. Show 5000 rows and you pay 5000 cells of memory, 5000 layout entries, and a multi-second build hitch. Lucent.Scroll instead keeps a bounded working set alive - only the cells that overlap the viewport (plus a small buffer) - and recycles them as you scroll. 5000 items render as roughly visible + 2×buffer live cells (about 9 for a five-row viewport), and the steady-state scroll path allocates zero bytes.

The whole thing is driven by one component, LcScrollView. It drives a uGUI ScrollRect by composition (never subclassing), self-builds its own viewport + content when dropped on a bare GameObject, and works with zero code: add the component, press Play, and a real 200-row sample list scrolls. Code always wins over the Inspector once you call a setter.

The data contract: you never hand it your list

Lucent.Scroll never stores your model. It asks an ILcScrollDataSource for a Count and to Bind data onto a recycled cell - and it only ever calls Bind for the handful of cells near the viewport, so binding is O(visible), never O(count). The fast path is a typed one-liner:

var view = gameObject.AddComponent<LcScrollView>();

// 5000 logical rows, but Bind() only fires for the ~9 cells on screen.
view.SetData<LcScrollTextCell>(5000, (i, cell) =>
    cell.SetContent("Row " + i, "subtitle " + i));

That wraps an LcListDataSource<TCell>. A Func<int> count overload handles lists whose length changes at runtime; implement ILcScrollDataSource directly for full control; or assign an LcScrollDataSource ScriptableObject for the no-code path.

The hot path: bounded work, zero steady-state allocation

LcRecycler is the heart of the asset. On every scroll event it asks the layout which index range is visible, releases the cells that left the window back to the pool, and fetches + binds + positions the cells that entered. The number of live cells never grows with the item count:

// LcRecycler.UpdateVisible -- the hot path (condensed)
public void UpdateVisible(float scrollOffset, float viewportMain)
{
    int count = _data.Count;
    _layout.GetVisibleRange(scrollOffset, viewportMain, out int first, out int last);
    first -= _buffer; last += _buffer;
    if (first < 0) first = 0;
    if (last > count - 1) last = count - 1;

    // ...release every live cell now outside [first,last] to its pool...

    for (int i = first; i <= last; i++)
    {
        if (_live.ContainsKey(i)) continue;          // already realised
        var cell = AcquirePool(i, out int type).Get();
        cell.CellType = type;
        cell.OnBind(i);
        _data.Bind(i, cell);                         // your binder, only for visible i
        Position(cell, i);
        _live[i] = cell;
    }
}

Why it stays at 0 GC. The loop uses struct Dictionary enumerators, a reused scratch List<int>, and no LINQ, closures, or boxing. The subtle part is positioning: a redundant write to a RectTransform's sizeDelta or anchors dirties the underlying uGUI Graphic and forces a per-cell mesh rebuild - the real GC source during scroll. So Position writes a property only when it actually changed:

if (rt.sizeDelta != size) rt.sizeDelta = size;
if (rt.anchoredPosition != pos) rt.anchoredPosition = pos;

For uniform cells the size and anchors are constant after warm-up, so only anchoredPosition ever moves - and the mesh is never rebuilt.

The pool and the cell lifecycle

LcCellPool wraps Unity's ObjectPool<T> with the collection check off. A released cell is deactivated and kept; a fetched cell is reactivated. Nothing is re-instantiated during scroll, which is exactly what TotalCellsCreated proves: it climbs while the pool warms, then stays flat forever. Every cell is an LcScrollCell with a reuse-aware lifecycle - OnBind(index) means "show item N now," OnRecycle() resets scale/alpha so a reused cell never carries stale animation state. The built-in LcScrollTextCell (a procedural Image + two uGUI Text fields, no TextMeshPro dependency) gives you the zero-prefab path.

Layouts: vertical / horizontal, list / grid / staggered

Geometry lives behind LcScrollLayout, which maps an index to a position and answers "which indices fall in this window?" without instantiating anything - so the recycler is layout-agnostic. Three layouts ship, on either axis (LcScrollAxis.Vertical or Horizontal):

  • LcScrollLinearLayout - a single track. Uniform extent by default, or per-cell variable sizes via SetVariableSizes(Func<int,float>), which builds an O(count) cumulative-offset cache once and finds the visible range by binary search - so even a 50,000-row list with every row a different height stays bounded.
  • LcGridLayout - fixed columns (vertical) or rows (horizontal): SetGridLayout(3).
  • LcStaggeredGridLayout - masonry: fixed columns, variable main sizes packed into the shortest column, via SetStaggeredGrid(crossCount, crossSpacing, mainSizeOf).

No-code juice: snapping, paging, pull, parallax, gamepad

Almost every behavior is a single Inspector toggle or a single setter. None of them touch the recycler's allocation budget:

view.SetSnapping(true, LcScrollSnapAlign.Center);   // settle on the nearest cell on drag end
view.SetPaging(true);                               // or snap by full viewport pages
view.SetPullToRefresh(true, thresholdPx: 80f);      // over-drag past an end fires OnPullToRefresh / OnLoadMore
view.SetLooping(true);                              // seamless wrap for pickers / endless feeds
view.SetAutoScroll(40f);                            // px/sec marquee; with Loop it wraps seamlessly
view.SetAnimation(LcScrollAnimationKind.Parallax, strength: 0.5f);
view.SetControllerNavigation(true);                 // gamepad d-pad / stick + arrow keys
view.ScrollToIndex(4200, LcScrollSnapAlign.Center, animated: true);  // jump deep into a virtualized list

Scroll-driven animation is a pure, testable function. LcScrollAnimator.Evaluate maps a cell's distance-from-viewport-centre in [0..1] to a scale/alpha pair for the FocusScale, Fade, and Depth presets; Parallax is distinct - a signed cross-axis positional shear via EvaluateCrossOffset (antisymmetric, so a cell dead-centre never moves). Every preset can be shaped by an AnimationCurve routed through Lucent.Motion's deterministic LcEasing. Applying it is alloc-free and writes only on change.

Gamepad navigation, two ways. When the view is the EventSystem's selected object, IMoveHandler.OnMove drives a focused cell. With SetControllerNavigation(true) it also polls the New Input System directly (d-pad / left stick + arrow keys, edge/repeat-gated). Either way, focusing an item that lives outside the live window auto-scrolls it into view first - FocusDelta is a pure axis/layout-aware mapping (a List steps along its scroll axis; a Grid steps by one within a line and by crossCount across lines), and OnFocusChanged reports the new index.

Animated data-ops without a rebind

After you mutate your data source, InsertItems, RemoveItems, and MoveItem animate the change instead of rebuilding. The trick is that the recycler re-keys the live cells in place rather than rebinding them - existing cells keep their instance and slide to their new slots, inserted cells pop in:

// after data.Insert(index, 3)
view.InsertItems(index, 3);   // existing visible cells slide down (same instances), 3 new cells pop in
// after data.Move(from, to)
view.MoveItem(from, to);      // the moved cell keeps its instance and slides to its new slot

The logical state is committed synchronously; the slide/pop is a brief cosmetic coroutine, skipped automatically when a scroll-driven scale animator already owns the cells. Drag-to-reorder (SetReorderable) and sticky section headers (SetSections) and frozen-header virtualized tables (SetTable) ride the same spine.

Theme-bound cells

Assign an LcScrollTheme asset (SetTheme / Inspector) to recolor the built-in text cells and drive spacing with zero scripting. LcScrollTheme.Apply resolves a cell's background, title/subtitle colours, and font sizes from the canonical global Lucent.Theme tokens (LcColorKeys.Surface, LcColorKeys.Text, LcNumberKeys.FontSizeTitle...) when a theme is active, falling back to the asset's serialized fields otherwise - so one theme switch recolours every scroll list, and an unconfigured asset still "just works." The cell setters write only on change, so theming never costs an extra mesh rebuild.

Prove the win

  • 5000 items, ~9 live cells. LiveCellCount stays bounded at visible + 2×buffer regardless of Count; TotalCellsCreated goes flat the moment the pool is warm - proof that scrolling recycles, never instantiates.
  • 0 bytes GC in steady-state scroll. Struct enumerators, a reused scratch list, no LINQ/closures/boxing, and write-on-change RectTransform updates that never trigger a uGUI mesh rebuild.
  • Binding is O(visible), not O(count). Your Bind delegate runs only for the cells near the viewport; ScrollToIndex(4200, ...) jumps deep into a 5000-row list without touching the other 4990.
  • Variable-height lists stay bounded. Per-cell sizes resolve through an O(count)-built cumulative cache and a binary-search visible range, so non-uniform rows don't degrade the recycler.
  • Certified in the shipped player. LcScrollSelfTest runs the real recycler in a built player under -selfTestScroll: 10,000 items must realise a bounded handful of cells (live > 0 && live < 100) with correct index binding, proving the path survives IL2CPP/AOT stripping - editor-green never certifies the device build.

Text: RTL/Arabic, BiDi, SDF/MSDF, and animated text without TextMeshPro

The Lucent.Text module is a complete Unicode text stack written in pure C#: bidirectional reordering, Arabic and complex-script shaping, line/word/grapheme segmentation, automatic font fallback, an SDF/MSDF glyph renderer that parses raw .ttf/.otf bytes itself, plus combat and animated text. It ships in two tiers. Tier 1 layers onto an existing TextMeshPro label as a drop-in ITextPreprocessor — correct Arabic/RTL with zero migration. Tier 2 (LcSdfText) renders to a MeshRenderer straight from font bytes with the Lucent SDF shader — no TextMeshPro, no native plugins, so it draws scripts and fonts a TMP atlas cannot.

Tier 1: one-call RTL, or a drop-in component

The shaping pipeline is a pure, allocation-cached function with no Unity dependency: Arabic contextual shaping (in logical order) → UAX #9 BiDi reordering → L4 mirroring → visual string. What the BiDi engine actually does. LcBidi.Resolve is a full UAX #9 implementation, not a heuristic "RTL fixer": paragraph level (P2–P3), explicit embeddings, overrides and directional isolates (X1–X9, including the X5a–X6a / FSI rules), weak types (W1–W7), paired-bracket and neutral resolution (BD16, N0–N2), implicit levels (I1–I2), and reordering with mirroring (L1–L4). That is why embedded Latin and numbers keep their order inside an RTL run, and why mixed Arabic/Hebrew/punctuation/brackets come out right where partial plugins fail.

Shaping: Arabic now, any complex script via the font

Tier 1's LcArabicShaper maps letters to their isolated/initial/medial/final presentation forms and builds lam-alef ligatures purely from Unicode joining data (no font file required); tashkeel marks are transparent to joining. Tier 2 goes further: LcOpenTypeArabicShaper and LcComplexShaper drive the font's own GSUB/GPOS tables — cmap lookup, joining features (isol/init/medi/fina), required and standard ligatures (rlig/liga), contextual alternates (calt), and GPOS mark placement. With Indic syllable reordering (LcIndicReorder) applied first, the same engine shapes Devanagari, Bengali, Tamil, Thai, and more, with the script auto-detected from the text.

// Variable fonts: interpolate an instance via fvar/gvar and re-rasterize.
if (label.HasVariableAxes) label.SetVariation("wght", 700f);

The same renderer composites COLR/CPAL colour emoji as stacked palette-tinted SDF layers (so a Segoe UI Emoji glyph draws with no bitmap tables), supports variable fonts through fvar/gvar interpolation, and offers a pixel-perfect snap mode (LcSdfPixelSnap) for crisp small text. Inline <gradient>/<outline>/<shadow>/<glow>/<bevel> style spans map to per-glyph vertex colours and material properties. Shaders load from Resources first so they survive build stripping on WebGL/IL2CPP.

Segmentation, wrapping, and font fallback

Line wrapping uses a full UAX #14 breaker (LcUax14LineBreaker.Analyze, implementing LB2–LB31 from baked Unicode property tables, with a lightweight LcLineBreaker for the hot path); LcLineWrapper.Wrap and LcTextLayoutEngine.Layout turn break opportunities and advance widths into positioned, aligned multi-line glyphs. Grapheme clusters (LcGraphemeBreak.ClusterStarts, UAX #29) keep a base and its marks together so the typewriter never splits an Arabic letter from its tashkeel; word and sentence breakers and a dictionary segmenter (for scriptio-continua like Thai) round out the set. LcFontFallback.SegmentRuns splits a code-point run into maximal runs that each resolve to the first font in a chain that actually has the glyph — one label can mix Latin, Arabic, CJK, and emoji even when no single font covers them all.

Animated and combat text

LcTextAnimator drives a typewriter that honours logical reading order through the cluster map (correct for RTL), plus tag-driven continuous behaviors (wave, shake, rainbow…), per-character entrances with easing, curved/arc and radial layout, and per-character sound — all by manipulating the TMP mesh in LateUpdate, with no custom shaders. A global ReduceMotion switch instantly settles every animator (an OS "reduce motion" analogue) while still firing reveal/word/named events so game logic is unaffected. For floating combat text, LcDamageNumbers.Spawn(value, worldPos) just works — a pooled, billboarded world-space (or canvas) manager that optionally combines rapid hits into one rising total.

  • No TextMeshPro dependency for Tier 2: LcSdfText.SetFont(byte[]) parses OpenType itself and renders via a MeshRenderer — no TMP font asset, no native code.
  • Real UAX #9, not a fixer: LcBidi.Resolve implements explicit levels, isolates, paired brackets, and L1–L4 reordering — the cases partial RTL plugins get wrong.
  • Logical-order animation after visual reordering: LcProcessedText.VisualToSource maps every displayed glyph back to its source index, so a typewriter reveals RTL text in true reading order.
  • Pure, cached, parallelizable core: immutable results are cached, and LcTextProcessor.ProcessBatch shapes hundreds of strings across all CPU cores.
  • Beyond Arabic: the font-driven LcComplexShaper + LcIndicReorder shape Devanagari, Bengali, Tamil, Thai and more from any font carrying GSUB/GPOS.

Components, Theming & Localization

Layer N8 is what most teams reach for first: a library of roughly two dozen no-code widgets, a design-token theme system that recolors the whole UI from one channel, and an I2-parity localization stack. All three sit on the same procedural spine as the rest of Lucent — widgets paint themselves with LcShape meshes and the built-in runtime font, so a fresh component renders correctly the instant it is added, with zero atlas, zero font import, and zero scene setup.

The widget library

The Lucent.Components assembly ships ~25 drop-in widgets, each a single MonoBehaviour whose serialized fields are the source of truth. Overlay/feedback widgets — LcDropdown, LcModal (dialog), LcToast, LcTooltip, LcContextMenu, LcTabs, LcAccordion/LcCollapsible, LcSegmentedControl, LcDatePicker (LcDate), LcTimePicker (LcTime), LcColorPicker, LcInputField, LcSlider, LcToggle, LcProgressBar, LcRatingStars, LcChip, LcBadge, LcSkeletonLoader, LcScrollbar, LcNavigationController — build their skin lazily in an EnsureBuilt() pass via the internal LcWidgetKit factory (LcWidgetKit.Shape(...), LcWidgetKit.Label(...)). Interaction rides the uGUI EventSystem through a tiny LcPointerRelay, so the widgets are backend-neutral and each exposes a UnityEvent (e.g. LcDropdown.OnValueChanged, LcModal.OnConfirmed, LcToast.OnDismissed) plus a plain code API.

Icons are fully procedural. LcIcon is an LcDrawable that tessellates one LcIconKind into a single batched mesh through LcIconGeometry.Draw; the LcIconKind catalogue holds 348 analytic glyphs (marks, arrows, media, files, devices, weather, commerce, shapes…), so every icon is resolution-independent and recolors or animates for free. LcAnimatedIcon adds spin/pulse and MorphTo(LcIconKind) transitions.

// No-code widgets are also fully drivable from script.
toast.ShowSuccess("Saved");            // queued, stacked, auto-dismissed (FIFO over _maxVisible)
dropdown.Value = 2;                    // clamps + raises OnValueChanged
icon.IconKind = LcIconKind.Settings;   // re-tessellates one batched mesh, no import

Just-works unconfigured. Every widget falls back to a calm dark default palette baked into LcWidgetKit (Surface, Primary, Text, …) and the engine's built-in LegacyRuntime.ttf, so it renders correctly with no theme and no font asset assigned. Bind an LcTheme and the same tokens resolve through it instead.

Design tokens & runtime theming

An LcTheme is a ScriptableObject holding six token tables — color, gradient, number, font, sprite, audio (the LcTokenCategory set) — each resolved by a stable string key. Keys are centralized as constants (LcColorKeys.Surface, LcColorKeys.Primary, LcNumberKeys.RadiusMd, LcGradientKeys.Primary, LcAssetKeys.FontDefault), so a single token swap restyles every surface that consumes it. Lookups are lazy-indexed and color resolution walks the explicit color table first, then the active LcPalette (color-blind-safe aware). Themes round-trip to BOM-less, deterministic JSON via ExportJson() / ImportJson(...) (gated on the raw text actually containing the required keys before any parse). Three built-in semantic themes — Dark, Light, HighContrast — are constructed entirely in code by LcSemanticThemes.Create(LcThemeId), each carrying the full color set, the shared numeric scale (DefaultNumbers()), a primary gradient, and an Okabe-Ito color-blind-safe palette variant.

Runtime theme switching flows through one static service, LcThemeManager. Theme-aware surfaces subscribe a single handler to LcThemeManager.Changed in OnEnable; switching the active theme — or editing the active theme in place — re-raises that one channel so every consumer re-resolves its tokens and re-tessellates. The hot path is allocation-free; the explicit batch entry point RebindAll(theme) returns the number of live surfaces it recolored (an assertable "applied" count) and, unlike the editor apply path, has no -batchmode guard, so a headless build or CI run gets a real count without walking the scene.

// Switch the whole UI to the built-in Light theme at runtime.
LcThemeManager.SetActiveTheme(LcThemeId.Light);

// Headless / CI batch re-bind returns the count of live surfaces recolored.
LcTheme theme = LcSemanticThemes.CreateHighContrast();
int recolored = LcThemeManager.RebindAll(theme);   // assertable "applied" count

// A widget resolves a semantic token, falling back to its just-works color.
Color32 surface = LcWidgetKit.Resolve(theme, LcColorKeys.Surface, LcWidgetKit.Surface);

Code precedence. LcThemeManager.Effective(bound) returns a consumer's own explicitly-bound theme when set and only falls back to the app-wide active theme (or literals) when code has not spoken — the "just-works" default never overrides a value set in code. A fresh domain reset leaves Active == null, so unconfigured surfaces keep their literal colors exactly as if the service did not exist.

Localization (term-DB, I2-parity)

The Lucent.Localization stack mirrors the term-database model of I2 Localization. An LcTermDatabase ScriptableObject holds a locale list and a table of LcTerms keyed by stable id ("ui.menu.play"); each term carries a Category, an LcTermKind (Text, or a per-locale Sprite/Audio/Font/Object asset swap), and a flat list of LcLocaleValues addressed by (locale, specialization, plural). Databases Merge by descending Priority (last-wins or keep-existing), and LcLocalizationIO round-trips the string tables through CSV/TSV (auto-delimited) and JSON. Resolution is a pure, deterministic function of its inputs: the LcLocalizationManager singleton walks a fallback chain (active → language-only → explicit chain → default) across all registered databases by priority, with optional auto-detect from Application.systemLanguage.

Plurals are CLDR-correct: LcPluralizer.Select dispatches by language subtag to the Arabic (all six categories), East-Slavic (ru/uk/be), Polish, Czech/Slovak, and French-like rule families, a no-inflection family for East-Asian locales, and a one/other fallback — all integer arithmetic, no culture or clock. LcFormat handles number/date/interpolation formatting per locale, and for RTL locales the LcLocalize component routes resolved strings through the Lucent.Text shaping/BiDi pipeline. LcLocalize binds a term to a UnityEngine.UI.Text, a TMP text (found via reflection — no TMP asmdef reference), or an LcText, supports {0}/{name} parameters and per-language font/size overrides, and re-applies automatically on every locale change.

LcLocalizationManager mgr = LcLocalizationManager.Instance;
mgr.RegisterDatabase(db);          // an LcTermDatabase (authored or CSV/JSON-imported)
mgr.SetActiveLocale("ru");         // raises LocaleChanged -> every LcLocalize re-resolves

// CLDR-correct plural form + {0}/{count} interpolation, number-formatted for the locale.
string msg = mgr.GetPlural("inbox.count", 5);   // ru selects the "few" form

// Or bind declaratively: drop LcLocalize on a UI.Text / TMP / LcText and set the term.
loc.TermId = "ui.menu.play";

Motion & layout

Widget animation is shared rather than reinvented. Lucent.Motion supplies LcEase (34 named curves) evaluated by LcEasing.Evaluate, an allocation-free LcTweenTimer (Create/Advance/Completed) plus LcTween/LcTweenVector2/LcTweenColor value tweens, a critically-damped LcSpring, and an authorable LcTweenSequence — this is what drives the modal scale-fade, the toast slide-in, and the accordion's height tween, all on unscaled delta time and pumpable headlessly via each widget's public Tick(float dt). Lucent.Layout adds flexbox-style arrangement (LcFlexLayout over LcFlexAxis), responsive scaling against breakpoints (LcResponsiveScaler), and content/aspect fitting (LcAutoSize, LcAspectMode).

  • 348 procedural icons in LcIconKind — analytic geometry, no atlas and no import step; every glyph scales crisply, recolors via a theme token, and animates for free.
  • One channel recolors everything. LcThemeManager.RebindAll() returns an assertable count of live surfaces recolored and runs in -batchmode with no scene walk — CI can verify a theme switch numerically.
  • I2-parity term model: terms keyed by id with category, specialization, and CLDR plural forms; multi-database merge by priority; CSV/TSV/JSON round-trip with raw-key gating on import.
  • Deterministic plurals for ar/ru/pl/cs/fr families plus a no-inflection East-Asian family, all pure integer arithmetic — same output on every platform, no culture dependency.
  • ~25 no-code widgets, each a single MonoBehaviour that builds its own procedural skin and works the instant it is added, recolors through LcTheme tokens, and is fully drivable from script.

SDK and No-Code Authoring

Lucent UI is code-first and no-code-convenient. Everything the editor hub can drop into a scene is a real component you could also create from a script, and every provider the system knows about lives behind a single typed seam: the LcRegistry. Third parties extend the asset by implementing ILcPlugin; the editor consumes the same registry through the Component Creator and Welcome windows. There is exactly one extension point, and the no-code layer always defers to it.

The registry seam: LcRegistry

LcRegistry (namespace Lucent.Sdk) is a typed register/resolve table with one category per provider family: shapes, effects, charts, render surfaces, chart scales, data transforms, and palettes. Each category exposes a Register* method (a stable id plus a factory delegate) and a TryCreate* resolver that constructs a fresh instance on demand. Palettes are immutable data, so they are stored and returned directly via RegisterPalette / TryGetPalette.

// Register (or override) a shape generator factory under a stable id.
public void RegisterShape(string id, Func<ILcShapeGenerator> factory);

// Construct a fresh shape generator by id; false (and null) when unknown.
public bool TryCreateShape(string id, out ILcShapeGenerator generator);

// Ordinal-sorted id set, for editor dropdowns and id-based creation.
public IReadOnlyCollection<string> ShapeIds { get; }

Two design choices make the registry safe to enumerate from editor UI. First, registration is last-wins: registering an id that already exists overwrites it, which is exactly how a plugin overrides a builtin. Second, enumeration is deterministic: every category is backed by a SortedDictionary<string, ...> keyed with StringComparer.Ordinal, so the id collections (ShapeIds, EffectIds, ChartIds, SurfaceIds, ScaleIds, TransformIds, PaletteIds) enumerate in a stable ordinal order rather than hash order. The aggregate Count sums every registered factory and palette across all categories.

The extension point: ILcPlugin + [LcPlugin]

A community or studio extension implements ILcPlugin and marks the class with the [LcPlugin] attribute. The interface is two members: a stable, prefixed LcPluginId that drives discovery ordering, and a Register(LcRegistry) method that adds the plugin's providers.

using Lucent.Sdk;
using Lucent.Render;   // ILcShapeGenerator

[LcPlugin]
public sealed class StudioPackPlugin : ILcPlugin
{
    public string LcPluginId { get { return "com.studio.lucentpack"; } }

    public void Register(LcRegistry registry)
    {
        // A new shape, or -- because registration is last-wins -- an
        // override of a builtin id such as "lc.shape.roundedRect".
        registry.RegisterShape("lc.shape.hexagon", () => new HexagonGenerator());
    }
}

Resolution anywhere in the asset goes through the one default registry exposed by LcSdk.Registry:

LcRegistry reg = LcSdk.Registry;
ILcShapeGenerator gen;
if (reg.TryCreateShape("lc.shape.hexagon", out gen))
{
    // 'gen' is a fresh instance, ready to emit geometry.
}

Discovery: bounded, attribute-gated, deterministic

LcPluginLoader.DiscoverAndRegister(registry) finds every marked plugin and calls its Register, returning the count successfully registered. Discovery is deliberately narrow: Discover() only inspects the Sdk assembly itself and assemblies that actually reference it, and within those only types carrying [LcPlugin] are considered. There is no arbitrary all-types reflection scan and no speculative instantiation of unmarked types. The resolved set is sorted by LcPluginId (ordinal) and de-duplicated by id, so registration order is stable across runs and across machines. A plugin whose Register throws is logged and skipped rather than aborting the whole load.

Composition root and startup

LcSdk.BuildDefault() is the composition root. It registers every builtin first -- all shape kinds via LcShapeGenerators.Registrations, the geometry effects, every LcChartKind generator, the seven positional scales, the data transforms, the three theme palettes, and the headless capture surface -- and only then runs LcPluginLoader.DiscoverAndRegister. Because plugins register last and registration is last-wins, a plugin can cleanly override any builtin id. LcSdk.PluginCount reports how many plugins were discovered.

LcSdkBootstrap.Init is a single [RuntimeInitializeOnLoadMethod(BeforeSceneLoad)] seam that touches LcSdk.Registry once to force the build. It is batch-guarded: under Application.isBatchMode it still builds the registry but performs no UI or logging side effects, so headless validation never hangs.

The no-code hub

The editor hub lives in Lucent.Hub.Editor and is pure code-driven IMGUI (no UXML). LcComponentCreatorWindow is the one-stop creator: it reads the live registry id lists (ShapeIds / EffectIds / ChartIds) into Shapes / Effects / Charts tabs, plus a curated Widgets tab, lets you pick a kind, and drops a configured component into the scene. LcWelcomeWindow is the first-run orientation screen (auto-opens once per machine), and LcHubMenu adds the cross-cutting Tools > Lucent UI entries (Hub, Documentation, Demo Scene) plus shared navigation to the Theme Designer and Localization windows -- routed through EditorApplication.ExecuteMenuItem so the hub never hard-references those assemblies.

Every "Create In Scene" action funnels through one shared helper, LcSceneScaffold.CreateUnder, so there is a single source of truth for Canvas + EventSystem provisioning, new-Input-System UI-module wiring, Undo registration, RectTransform setup, selection, and ping. ResolveCanvas reuses an existing Canvas (or creates a ScreenSpaceOverlay one) and EnsureEventSystem reuses or creates an EventSystem with an InputSystemUIInputModule resolved by name -- so running an action twice never litters duplicates.

Defer-to-code precedence

The no-code layer is a thin convenience over the code-first runtime, and it defers to code at every boundary:

The lower layers (L0-L5) construct their builtins directly; the Sdk only registers them for enumeration and id-based creation. Nothing in those layers references Lucent.Sdk -- there is no upward dependency edge. The registry is a lookup index over code, not the authority for it.

Inside the hub, component types are resolved by name via Type.GetType("Ns.Type, Assembly") with a loaded-assembly fallback, so the window degrades gracefully if a runtime area is absent. Configuration is best-effort: TrySetKindFromId maps a registry id's last segment (for example lc.shape.roundedRect to RoundedRect) onto a public *Kind enum property, and a missing property or mismatch is a logged no-op, never a hard failure. Every scene mutation is deferred out of the IMGUI layout pass through EditorApplication.delayCall, and the type is re-resolved at click time. The hub authors the same components you would author in code, then steps out of the way.

  • One seam, typed: all builtins and all third-party depth enter through LcRegistry via Register* / TryCreate* -- a 36th chart type or a custom effect needs no core changes.
  • Deterministic enumeration: ordinal-sorted SortedDictionary id sets and id-sorted plugin discovery mean editor dropdowns and registration order are identical across runs and hosts.
  • Bounded, safe discovery: only assemblies referencing the Sdk and only [LcPlugin]-marked types are inspected -- no all-types reflection scan, no speculative instantiation.
  • Graceful degradation: a plugin whose Register throws is logged and skipped; DiscoverAndRegister returns only the count that succeeded.
  • Batch-safe: LcSdkBootstrap builds the registry headlessly with no UI side effects, and LcWelcomeWindow.AutoOpenOnce bails on its first line under Application.isBatchMode, so CI validation cannot hang.
  • No scene litter: LcSceneScaffold.CreateUnder reuses an existing Canvas and EventSystem, registers Undo, and pings the result -- repeat clicks stay clean.

Samples & the interactive demo

Lucent ships two distinct things to learn from, and they are built to opposite goals. The samples (Assets/Lucent/Samples) are eight tiny, single-feature teaching scenes — open one, press Play, read the one small script next to it. The interactive demo (Assets/Lucent/Demo, the Lucent.Demo assembly) is a single runtime-built showcase app with one module per niche, a persistent navigator/stage/controls chrome, and a live "prove-the-win" HUD — and it is WebGL-buildable end to end. Both build their UI from code against the real shipped Lucent components, so the saved .unity assets stay tiny and deterministic.

The samples: one feature per scene

Each sample is a single MonoBehaviour that implements ILcSample and constructs its example in Start(). The contract is deliberately minimal — a sample exposes exactly one member, SceneName (the scene file stem), and is otherwise free to teach its one feature however reads best. The shared plumbing is one static helper, LcSampleStage, whose CreateRoot self-heals a ScreenSpaceOverlay Canvas and an EventSystem (new Input System when present, legacy module otherwise) and hands back a full-stretch root to build under:

namespace Lucent.Samples
{
    using UnityEngine;
    using Lucent.Render;   // LcShapeKind, LcCorners, LcStrokeAlign (shared spine types)
    using Lucent.Shapes;   // LcShape, LcShapeStyle (the feature taught here)

    public sealed class S01_Shapes : MonoBehaviour, ILcSample
    {
        // Pure literal: the scene builder reads it right after AddComponent, before Start.
        public string SceneName => "01_Shapes";   // -> Scenes/01_Shapes.unity

        void Start()
        {
            // Self-heals a Canvas + EventSystem; returns a full-stretch RectTransform.
            RectTransform root = LcSampleStage.CreateRoot(SceneName);

            var go = new GameObject("RoundedRect", typeof(RectTransform));
            go.GetComponent<RectTransform>().SetParent(root, false);
            LcShape rrect = go.AddComponent<LcShape>();
            rrect.ShapeKind = LcShapeKind.RoundedRect;
            rrect.SetCornerRadii(new LcCorners(56f, 14f, 56f, 14f)); // TL, TR, BR, BL
            rrect.SetFillColor(new Color32(99, 102, 241, 255));      // indigo
            rrect.SetStroke(true, 4f);                                // 4px inner outline
        }
    }
}

The eight scenes live under Assets/Lucent/Samples/Scenes, one per niche: 01_Shapes, 02-Effects-Glass, 03-Buttons, 04-Charts, 05-Scroll, 06-Text, 07-Components, 08-Theming-Localization. They are hand-opened teaching scenes and intentionally are not added to Build Settings. To open one: build the scenes (below), open Assets/Lucent/Samples/Scenes/<NN_Name>.unity, press Play, then read the matching S0N_*.cs file — the teaching comments are the product.

You do not author scenes by hand. LcSampleSceneBuilder (in Lucent.Samples.Editor) reflects over every ILcSample in the assembly and emits one scene each — a Camera (solid dark clear) + a ScreenSpaceOverlay Canvas + an EventSystem + the bare sample component. The interactive demo: one module per niche

The demo is a persistent shell plus pluggable modules. LcDemoController is one MonoBehaviour that builds the whole chrome at runtime — a top title bar, a left feature navigator, a center stage, a right per-feature controls panel, and a bottom prove-the-win HUD — and self-provisions its own Canvas + new-Input-System EventSystem so it runs from a bare GameObject. Each feature page is a class implementing ILcDemoModule; LcDemoRegistry.Discover() reflects over the Lucent.Demo assembly, finds every public module with a parameterless constructor, de-dupes by type, and sorts by Title (the "N1 "..."N9 " tag makes navigator order deterministic). Modules never touch the chrome and never see each other — so all nine were authored in parallel against this one contract:

public sealed class LcModuleN1Shapes : ILcDemoModule
{
    public string Title => "N1 - Procedural Shapes";
    public string Niche => "One mesh, one draw call, zero textures.";

    // stage = empty center canvas; controls = empty vertical auto-stack panel.
    // Build everything from the REAL Lucent components via the DemoUI factory.

    {
        LcShape card = ui.Panel(stage, 20f, elevated: true, "N1Card");

        ui.Button(controls, "Cycle shape", () => hero.ShapeKind = LcShapeKind.Star);
        ui.Slider(ui.Field(controls, "Stroke width"), 0f, 14f, 3f,

    // 1..6 short "metric: value" strings rendered into the bottom HUD.
    public string[] ProveTheWin() => new[]
    {
        "1 draw call (fill+stroke)", "0 textures sampled", "0 B/frame GC", "13 procedural families"

Every widget comes from the DemoUI factory, which wraps the genuine components (LcShape surfaces, LcButton, LcSlider, LcToggle) and the demo design tokens behind a tiny shared surface (Panel, Surface, Column, Row, Button, Slider, Toggle, Field, Label). FAQ & Troubleshooting

Practical answers to the questions that come up most often when dropping Lucent UI into an existing project. If something here does not match what you see, mail the exact symptom, your Unity version, and your active render pipeline to szekipapa77@gmail.com.

Which input backend does it need — old, new, or both?

Either, including both at once. The interactive widgets (LcButton and friends) derive from uGUI's Selectable and implement the standard IPointerClickHandler / ISubmitHandler interfaces, so they are driven by whatever input module sits on your EventSystem — the legacy StandaloneInputModule or the new InputSystemUIInputModule. Both work with zero changes.

Where Lucent reads a raw pointer directly (chart pan/zoom, hover), it goes through one device-agnostic bridge, LcInputAdapter.Read(), which returns an LcPointerState. It prefers the new Input System's Mouse.current when the package is present, falls back to the legacy UnityEngine.Input manager otherwise, and returns a zeroed (no-op) state when neither backend is available — so it imports and runs no matter how Project Settings > Player > Active Input Handling is set.

Detection is automatic. The new-Input-System branch is guarded by the LUCENT_INPUTSYSTEM define, which the asmdef raises only when com.unity.inputsystem (1.0.0+) is installed. The legacy branch is guarded by Unity's own ENABLE_LEGACY_INPUT_MANAGER. You never edit a define by hand.

Do I need URP or HDRP? I'm on Built-in.

No. URP and HDRP are optional. The core asset never references either package; the render spine draws procedural meshes through the standard uGUI canvas material, so the vast majority of Lucent — shapes, buttons, charts, scroll, text, theming, effects — is pipeline-agnostic and runs on Built-in out of the box. At runtime the pipeline is sniffed from GraphicsSettings.defaultRenderPipeline via LcCapabilities.ActivePipeline, which returns one of LcPipeline.BuiltIn, Urp, Hdrp, or Unknown. Pipeline-specific code lives in isolated sub-assemblies (Lucent.Effects.Urp behind LUCENT_URP, Lucent.Effects.Hdrp behind LUCENT_HDRP) that compile out entirely when the package is absent.

The one feature that benefits from a live scene-color grab is the N3 blur / liquid-glass hero. It works on all three pipelines from one piece of dual-Kawase math: Built-in records the down/up pyramid through an LcBlurBuiltInManager + CommandBuffer (no SRP package required), URP through a RenderGraph ScriptableRendererFeature, and HDRP through a CustomPass. Where no grab provider exists, LcGlass.IsSupported reads false and the panel degrades to a flat tinted translucent fill — never a broken (magenta) shader, never a null-ref.

Do I have to install or import TextMeshPro?

No, on both counts. TextMeshPro is merged into com.unity.ugui (it ships Unity.TextMeshPro.asmdef under com.unity.ugui/Runtime/TMP), so there is no separate package to add and no "Import TMP Essentials" step gating Lucent text. More importantly, Lucent does not require TMP at all: every built-in label rides Unity's bundled LegacyRuntime.ttf font, and the Tier-2 text renderer LcSdfText parses raw .ttf/.otf bytes itself and draws through a plain MeshRenderer — no TMP font asset, no native plugin. If you do use TMP, the Tier-1 LcText component wires itself in as an ITextPreprocessor for correct Arabic/RTL with zero migration.

Why is my whole element "one draw call," and when does that break?

Every Lucent visual is an LcDrawable that tessellates its shape, gradient/pattern/sprite paint, stroke, and its sibling geometry effects (shadow, glow, outline) into a single pooled LcDrawList in painter order, which becomes one CanvasRenderer mesh. Identically-configured elements share one cached material and batch together. The batch is only broken by the things that always break uGUI batching: a non-Normal blend mode (each distinct one gets exactly one extra surface), a child Text/TMP overlay (the one intrinsic extra draw), an interleaved non-Lucent graphic that forces a different material, or a Mask/RectMask2D boundary. Keep effects on Normal blend and elements on the shared material and the batch holds.

My glass / blur panel just shows a flat tint. Why?

That is the intentional graceful-degradation path, not a bug. LcBlurPanel/LcGlass sample a global _LucentBlurTex that a per-pipeline backend publishes from the scene color. If no live scene-color grab is available — for example Built-in with no grab provider attached yet, or a headless run — LcCapabilities.SupportsSceneColorGrab is false and the panel renders a flat tinted translucent fill instead of magenta. To get the real frosted look, make sure a camera is rendering the scene behind the panel: on URP add the LcBlurFeature to your renderer; on Built-in the LcBlurBuiltInManager auto-attaches its command buffer while a panel is alive. Blur work is ref-count-gated, so it only runs while at least one panel is active.

How does the scroll list stay at 0 GC with thousands of rows?

LcScrollView never stores your model and never instantiates per row. LcRecycler keeps only visible + 2×buffer cells alive, pools the rest, and its hot path uses struct enumerators, a reused scratch list, and no LINQ/closures/boxing. The one subtle source of scroll-time GC — a redundant RectTransform write that dirties a uGUI Graphic and forces a mesh rebuild — is avoided by writing sizeDelta/anchoredPosition only when they actually change. If you do see allocation during scroll, it is almost always your own Bind delegate (string concatenation, boxing, LINQ) rather than the recycler; the recycler calls Bind only for the handful of on-screen cells.

Can it run headless / in CI without hanging?

Yes — this is a first-class design constraint. The composition root LcSdkBootstrap still builds the registry under -batchmode but performs no UI or logging side effects, and the editor Welcome window bails on its first line in batch mode, so validation never blocks on a dialog. The charts layer exports to PNG/SVG/CSV/JSON on the CPU with no GPU, deterministically (invariant culture, no clock, no GUIDs). The sample and demo scene builders self-exit with a clean status code (0 success / 13 failure), and LcThemeManager.RebindAll(theme) returns an assertable count of recolored surfaces without walking the scene — so CI can gate on real numbers.

What are the actual package dependencies?

Exactly one is mandatory: com.unity.ugui (2.0.0+). Everything else lights up features and is gated behind a versionDefine, so a missing package drops its code out instead of breaking compilation: com.unity.inputsystem (recommended → LUCENT_INPUTSYSTEM), com.unity.render-pipelines.universal / .high-definition (optional → LUCENT_URP / LUCENT_HDRP, for the blur/glass hero), and com.unity.modules.uielements (optional → LUCENT_HAS_UITOOLKIT, for the UI Toolkit surface). Target Unity 6000.0.62f1 or newer.

  • Input-backend-agnostic: old, new, or both — widgets ride the EventSystem and raw reads route through LcInputAdapter, down to a zeroed no-op.
  • Built-in is a first-class target: the core is pipeline-agnostic; URP/HDRP only enrich the blur/glass hero and compile out when absent.
  • No TextMeshPro requirement: built-in labels use LegacyRuntime.ttf and LcSdfText parses font bytes itself.
  • Predictable batching: one LcDrawable = one mesh = one draw call until a known batch-breaker (non-Normal blend, text overlay, mask) intervenes.
  • Headless/CI-safe by design: batch-guarded bootstrap, GPU-free deterministic export, and assertable applied-counts.
